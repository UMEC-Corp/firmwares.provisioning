    1. Persistent data model

    1. Persistent Data Model

    Persistence layer хранит только bootstrap/cloud state. Это хороший проектный выбор: transport flags, live connectivity bits и transient workers не попадают в NVS. Благодаря этому reboot возвращает system intent, а не случайный runtime residue.

    The persistence layer stores only bootstrap and cloud state. That is a good project decision: transport flags, live connectivity bits, and transient worker state do not leak into NVS. As a result, reboot restores system intent rather than random runtime residue.

    2. Time validity как gate для telemetry

    2. Time Validity as a Telemetry Gate

    Телеметрия не публикуется, пока has_valid_time() не вернет true. В текущем коде валидным считается Unix time не раньше 2017-01-01. Это защищает облако от заведомо мусорных timestamps сразу после boot.

    Telemetry is not published until has_valid_time() returns true. In the current code, valid time means a Unix timestamp no earlier than 2017-01-01. This protects the cloud from obviously bad timestamps right after boot.

    Boot
-> restore time from NVS if possible
-> allow telemetry only if restored/synced time is sane
-> later resync with SNTP
-> persist a new good snapshot back into NVS

    3. Remote config refresh

    3. Remote Config Refresh

    При наличии сети и заданного configUrl background maintenance task примерно раз в сутки вызывает fetch_and_store_remote_config(). Функция не просто парсит ответ, но и строит human-readable change log для логов, перечисляя какие именно URL changed.

    When network connectivity exists and configUrl is set, the background maintenance task calls fetch_and_store_remote_config() roughly once per day. The function not only parses the response but also builds a human-readable change log describing which URLs changed.

    4. OTA execution path

    4. OTA Execution Path

    OTA route starts from MQTT command upgrade. Дальше firmware открывает HTTP client, получает next update partition, начинает OTA session через esp_ota_begin(), читает бинарный поток блоками по 1024 байта и пишет их через esp_ota_write(). Только после успешного esp_ota_end() вызывается esp_ota_set_boot_partition().

    The OTA route starts from the MQTT upgrade command. The firmware then opens an HTTP client, obtains the next update partition, begins an OTA session with esp_ota_begin(), reads the binary stream in 1024-byte blocks, and writes those blocks via esp_ota_write(). Only after a successful esp_ota_end() does it call esp_ota_set_boot_partition().
